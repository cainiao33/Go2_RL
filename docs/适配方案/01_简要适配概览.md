# 冠军方案适配概览（一页纸）

## 背景

- **冠军方案**：`go2_rl_gym-master`，基于 Isaac Gym + rsl_rl，使用 CTS（Teacher-Student）/ MoE 架构
- **当前赛题**：腾讯开悟平台 KaiwuDRL + Isaac Lab，Go2 四足机器人自主导航运控
- **核心差异**：冠军方案的环境、域随机化、观测、训练调度都在本地 Isaac Gym 内完成；本届需适配到平台提供的 Isaac Lab 封装环境中

---

## 赛题核心要求

| 模式 | 评分公式 | 最高优先级 |
|------|----------|------------|
| Standard | 0.4×距离 + 0.2×时间 + 0.2×能耗 + 0.2×姿态 | 走穿地形（距离 ≥ 3.9m） |
| Track | 完成率 × (0.4×时间 + 0.4×姿态 + 0.2×能耗) | **先完成**，再快、再稳 |

---

## 适配核心思路

```
冠军方案能力拆解：
┌─────────────────────────────────────────────────┐
│ 1. 鲁棒 locomotion（PPO + 丰富域随机化 + 课程）  │ ← 可直接借鉴
│ 2. Teacher-Student 蒸馏（CTS）                   │ ← 需改造
│ 3. MoE 多专家路由                                │ ← 可选进阶
│ 4. 导航能力（command curriculum + dynamic sigma） │ ← 需结合 track 目标点
└─────────────────────────────────────────────────┘
```

**可直接迁移的**：
- 奖励函数设计思路（权重比例、动态 sigma、action smoothness）
- PPO 超参经验（adaptive KL、clip、GAE λ）
- 课程学习策略（command range 逐步扩大、地形难度递进）

**需要改造的**：
- CTS 架构 → 平台 policy 已有 height_scan，不再是 blind student
- 域随机化 → 平台仅开放 friction / push，其余需用 reward 替代鲁棒性
- 环境交互 → 必须适配 KaiwuDRL workflow 接口
- 导航 → 需自行从 `env.goal_positions` / `nav_scanner` 构造特征

**不可直接用的**：
- Isaac Gym 底层 API（gymapi、gymtorch）
- 机器人 URDF/资产配置
- motor strength / PD gain / action delay 随机化

---

## 推荐路线（分阶段）

| 阶段 | 目标 | 时间占比 |
|------|------|----------|
| Phase 1 | Standard 模式：调优 PPO baseline，借鉴冠军 reward + 课程 | 40% |
| Phase 2 | Track 模式：加入目标点观测 + 导航 reward | 30% |
| Phase 3 | 进阶：尝试 CTS 蒸馏或 history buffer 提升泛化 | 20% |
| Phase 4 | 精调：能耗/姿态优化，评估调参 | 10% |

---

## 一句话结论

> 冠军方案的**训练经验和奖励设计**高度可迁移，但**环境控制层和模型架构**需要重新适配到平台约束下。核心收益来自 reward 工程和课程策略的借鉴，而非代码的直接复制。
