# 总体审查汇总与下一步建议

## 总体结论

当前 `docx/适配方案` 下的 5 份文档整体方向正确：

- 不建议直接照搬冠军 Isaac Gym 代码
- 推荐以当前 `agent_ppo` 为主线
- 借鉴冠军方案的 reward、课程学习和训练经验
- 先 Standard locomotion，再 Track navigation
- CTS/MoE 暂缓

但文档中存在几类共性风险，需要在真正动手改代码前修正。

---

## 一、最高优先级修正项

### 1. Track 观测维度必须统一

当前文档中出现了多种说法：

- 301 + 3 = 304
- 301 + 5 = 306
- 301 + 6~8

建议统一为：

```text
policy_obs_dim = 306 = 301 + 5
critic_obs_dim = 321 = 316 + 5
```

推荐 goal features：

```text
[dx, dy, distance, cos(goal_yaw), sin(goal_yaw)]
```

如果拿不到 robot yaw，第一版不要做本体系旋转，先用世界系相对目标向量。

### 2. 为预训练兼容预留 goal feature

如果 Standard 是 301 维，Track 是 306 维，则 Standard 模型无法直接完整加载到 Track 模型。

建议：

```text
Standard obs = default_obs(301) + zeros(5)
Track obs    = default_obs(301) + goal_features(5)
```

这样 Standard 和 Track 都用同一个 Actor 输入维度 306，便于无缝继续训练。

### 3. 不要一开始大幅改 reward 权重

文档中一些权重偏激进，例如：

- `reach_goal = 30.0`
- `dof_pos_limits = -5.0`
- `joint_torques = -2e-4`
- `friction_range = [0.2, 2.5]`
- `max_push_vel_xy = 1.5`

建议第一版采用保守策略：

```text
先跑通默认 PPO → 小幅调 reward → 再逐步增强复杂 reward/随机化
```

### 4. 新增 reward 要先验证可访问状态

高风险 reward：

- `feet_air_time`
- `base_height`
- contact-force based reward
- 二阶 `action_smoothness`

这些依赖底层接触、历史动作、刚体状态，平台是否暴露不确定。应先实现低风险 reward：

- `stand_still`
- `hip_to_default`
- `goal_direction`
- `navigation_progress`

---

## 二、技术路线建议

### 推荐路线

```text
P0: 保持 baseline PPO 跑通 Standard
P1: 统一输入维度，预留 goal feature
P2: Standard reward/curriculum 小幅调优
P3: Track goal feature + dense navigation reward
P4: proprio history buffer
P5: 能耗/姿态精调 + checkpoint 筛选
```

### 暂缓路线

```text
CTS / MoE / 多专家 / 接触类复杂 reward / nav_scanner 深度使用
```

原因：

- 当前 policy 已有 height_scan，CTS 收益不确定
- MoE 调试成本高
- nav_scanner 接口和数据格式需要验证

## 四、关键判断标准

### Standard 阶段是否可以切 Track？

建议满足：

```text
各地形 distance_score > 0.7
abnormal_count 明显低于 completed_count + timeout_count
pyramid_stairs / pyramid_stairs_inv 不再频繁摔倒
```

### Track 是否在学？

看：

```text
completed_count 是否从 0 增加
navigation_progress reward 是否上升
timeout_count 是否逐渐下降
abnormal_count 是否可控
```

### 是否需要上 history buffer？

如果出现：

```text
复杂地形上动作抖动
下楼梯姿态恢复差
速度跟踪滞后
同一地形表现不稳定
```

则优先做 proprio history，而不是 CTS。

## 五、最终建议

当前 5 份方案可以作为方向文档保留，但真正实施前建议先完成以下三件事：

1. **统一 Track/Standard 输入维度，解决预训练兼容问题**
2. **把 reward 和域随机化建议分成保守版、增强版、实验版**
3. **补充接口验证步骤，特别是 goal_positions、robot root position、nav_scanner、contact 状态**

一句话：

> 当前方案方向对，但第一版实现要更保守、更可验证。先做 PPO + 固定 goal feature + dense navigation reward，跑出可评估基线后，再逐步引入冠军方案中的复杂技巧。
