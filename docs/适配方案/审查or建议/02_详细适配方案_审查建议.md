# 02_详细适配方案 审查/建议

## 总体评价

这份文档覆盖面最广，已经具备后续实施路线图的雏形。但其中有几类内容需要谨慎：一是部分代码示例依赖的属性未确认；二是一些 reward 名称虽然合理，但平台是否能从 `RewardProcessBase` 中访问到对应状态需要验证；三是直接新增 `TrackConfig`、改输入维度会影响模型加载和预训练迁移，需要更明确的兼容策略。

## 可保留的核心建议

- **保留 baseline PPO 主体**：合理。现有 `agent_ppo` 已经实现完整 PPO，没必要一开始重写 `agent_diy`。
- **Track 追加 goal feature**：必要。赛题文档明确 `default_observation()` 不包含目标点信息，Track 需要自行拼接。
- **Standard → Track 预训练**：合理。先训练通用 locomotion，再迁移到导航任务。
- **新增 track 配置文件**：合理。Standard 和 Track 的地形、episode 长度、奖励权重都不同，应拆配置。
- **新增 monitor 面板**：必要。否则新增 reward 难以调参。

## 重点风险与修正建议

### 1. 模型输入维度改动会破坏预训练加载

文档建议 Standard 使用 301 维，Track 拼接 goal 后变成 306 或 304 维。这会导致 Standard 训练出的 Actor 第一层权重无法直接加载到 Track 模型。

建议采用以下策略之一：

#### 策略 A：从一开始预留 goal feature 维度

Standard 阶段也输出固定维度，例如 306 维：

```text
Standard obs = default_obs(301) + zeros(5)
Track obs    = default_obs(301) + goal_features(5)
```

优点：Standard 模型可无缝作为 Track 预训练模型。

#### 策略 B：实现部分加载

只加载除 Actor 第一层外的参数，或手工把原 301 维权重拷贝到新 306 维权重前 301 列，其余列初始化为 0。

优点：不用改 Standard 观测。

缺点：代码更复杂，容易出错。

**建议优先用策略 A。**

### 2. goal feature 维度前后不一致

文档中有：

- “+6~8 维”
- 示例是 `301 + 5 = 306`
- 环境选择文档中又是 `301 + 3 = 304`

建议统一为 5 维或 6 维：

```text
goal_features = [goal_local_x, goal_local_y, distance, cos(goal_yaw_error), sin(goal_yaw_error)]
```

推荐 5 维，简单稳定。若要加 nav scanner 聚合特征，可再额外加：

```text
nav_clearance_front, nav_clearance_left, nav_clearance_right
```

最终可设为：

```text
policy_obs_dim = 301 + 5 = 306
critic_obs_dim = 316 + 5 = 321
```

### 3. `self.env` 访问方式需要实际验证

文档示例中使用：

```python
self.env.goal_positions
self.env.scene["robot"].data.root_pos_w
```

但 `PolicyObservationProcess` 中是否直接持有 `self.env`，以及 robot 路径是否叫 `"robot"`，需要读 `tools` 或实际打印确认。建议先写成审慎方案：

- 先确认 `ObservationProcess` 基类暴露哪些属性
- 优先使用文档明确给出的 `env.goal_positions`、`env.goal_yaw`
- `nav_scanner` 先只做备选，不在第一版实现中依赖

### 4. 新增 reward 不一定能访问底层状态

文档建议新增：

- `action_smoothness`
- `feet_air_time`
- `stand_still`
- `base_height`
- `hip_to_default`

其中风险最大的是：

- `feet_air_time`：需要 feet contact 状态和历史接触缓存
- `base_height`：需要 terrain height 或 body/root state
- `action_smoothness`：需要 `last_last_actions`

当前文档没有确认 `RewardProcessBase` 是否暴露这些属性。建议分级实施：

#### 第一优先级：容易从 obs/action 构造

- `stand_still`
- `hip_to_default`
- `forward_velocity`（如果示例中已有）
- 简化版 `action_smoothness`（如有 last_action）

#### 第二优先级：需要 env 内部状态

- `feet_air_time`
- `base_height`
- 完整二阶 action smoothness

#### 第三优先级：暂缓

- 任何依赖 contact forces 或 rigid body states 的 reward，除非确认平台暴露。

### 5. PPO 超参建议偏激进

文档建议：

```text
num_envs = 4096
num_steps_per_env = 24
lr = 1e-3
```

风险：当前 baseline 默认 `lr=3e-4`、`num_steps_per_env=48`，示例代码有稳定性防护但平台环境复杂，直接把 lr 拉到 `1e-3` 可能导致 early collapse。

建议采用更稳的阶梯：

| 方案 | num_envs | num_steps_per_env | lr | 适用阶段 |
|------|----------|-------------------|----|----------|
| 稳妥 baseline | 2048 | 48 | 3e-4 | 第一次跑通 |
| 提速版 | 4096 | 32 | 5e-4 | 跑通后 |
| 激进版 | 4096 | 24 | 1e-3 | 确认 KL 稳定后 |

### 6. TOML 新 reward 名称必须和函数名严格对应

如果 TOML 中写：

```toml
[rewards.action_smoothness]
```

则代码中应有：

```python
def _reward_action_smoothness(...)
```

同时要在 `monitor_builder.py` 中添加对应指标，否则难以判断 reward 是否生效。

### 7. Track reward 权重可能过强

`reach_goal = 20~30` 的方向是对的，但如果稀疏奖励过强且前期无法到达，会对训练贡献很小。建议搭配 dense reward：

- `navigation_progress`：距离缩短奖励
- `goal_direction`：朝目标方向速度分量
- `timeout_penalty`：超时轻微惩罚
- `abnormal_penalty`：摔倒强惩罚

第一版不要只依赖 `reach_goal`。

## 建议后的实施优先级

| 优先级 | 内容 | 原因 |
|--------|------|------|
| P0 | 保持 301/316 baseline 跑通 Standard | 建立基线 |
| P1 | 新增固定维度 goal placeholder，统一 306/321 | 保证预训练兼容 |
| P1 | 新建 Track TOML，加入 goal dense reward | Track 必需 |
| P2 | 新增 stand_still / hip_to_default / action_rate 加强 | 低风险 reward |
| P2 | 调整地形比例和 command ranges | 无需改代码 |
| P3 | feet_air_time / contact 类 reward | 需验证可访问状态 |
| P4 | CTS/MoE | 成本高，延后 |

## 结论

该文档很有实施价值，但建议把“可直接改”的内容和“需验证平台接口”的内容分开。最重要的修正是：**Track 观测维度必须统一并考虑预训练加载兼容性**。建议先采用“Standard 也补零 goal feature”的固定输入维度方案，降低后续迁移成本。
