# 01_简要适配概览 审查/建议

## 总体评价

这份文档定位清晰，适合作为“冠军方案能不能迁移”的快速判断材料。核心判断基本正确：`go2_rl_gym-master` 的价值主要在 reward、课程学习、训练经验，而不是直接复制 Isaac Gym 环境代码。

## 判断基本正确的部分

- **工程差异判断准确**：冠军方案是 Isaac Gym + rsl_rl，本项目是 KaiwuDRL + Isaac Lab 平台适配，两者训练调度和环境封装差异很大。
- **不可直接迁移项列得合理**：`gymapi`、`gymtorch`、URDF、PD 参数、action delay 等底层能力在当前平台中不可直接操作。
- **分阶段路线合理**：先 Standard locomotion，再 Track navigation，最后精调能耗/姿态，符合赛题评分结构。
- **Track 优先级判断正确**：Track 分数有完成率乘子，完成率确实优先于时间、姿态和能耗。

## 需要修正或补充的点

### 1. “平台仅开放 friction / push”表述略窄

文档写道平台域随机化仅开放 `friction / push`，但赛题文档中还明确有：

- `domain_rand.enable_domain_rand`
- `domain_rand.randomize_friction`
- `domain_rand.friction_range`
- `domain_rand.push_robots`
- `domain_rand.push_interval_s`
- `domain_rand.max_push_vel_xy`
- `noise.add_noise`

严格说，观测噪声不属于 domain_rand 段，但它也是训练鲁棒性来源。建议改为：

> 当前平台显式开放的随机化主要是 friction、push 与 observation noise，冠军方案中的质量、COM、PD gain、motor strength、action delay 等未直接暴露。

### 2. “导航能力”不要只写 command curriculum

冠军方案的 `command curriculum` 更偏 locomotion 命令泛化，不等价于本赛题 Track 中的目标点导航。本赛题导航关键是：

- `env.goal_positions`
- `env.goal_yaw`
- `nav_scanner`
- `reach_goal`
- progress/direction reward

建议把第 4 点改成：

> 导航能力：冠军方案的 command curriculum 可借鉴，但 Track 模式必须额外引入 goal-relative features 与导航进度奖励。

### 3. Phase 3 的 CTS 优先级建议再降低

原文给 Phase 3 分配 20% 时间给 CTS/history buffer。根据当前平台 policy obs 已有 301 维，其中包含 256 维 height scan，CTS 的收益不确定，且实现成本高。建议把 Phase 3 改成：

- 优先：history buffer 或 GRU
- 次选：CTS
- 不建议早期做 MoE

## 建议修改后的推荐路线

| 阶段 | 目标 | 建议占比 |
|------|------|----------|
| Phase 1 | Standard locomotion baseline 调优 | 45% |
| Phase 2 | Track goal feature + navigation reward | 30% |
| Phase 3 | History buffer / GRU / reward 精调 | 15% |
| Phase 4 | 评估回归、能耗姿态精调、checkpoint 筛选 | 10% |

## 风险提示

- **风险 1**：如果过早引入 Track，可能因为基础步态不稳导致 reward 稀疏、训练不收敛。
- **风险 2**：如果只看冠军方案结构而忽略赛题 scoring，可能优化错目标。Standard 要先走远，Track 要先完成。
- **风险 3**：不能把冠军方案的 privileged obs 维度、地形高度点数、域随机化参数直接照搬到当前平台。

## 结论

该文档可以保留作为总览，但建议补充“观测噪声也是鲁棒性来源”“Track 导航必须引入目标点/扫描特征”“CTS 优先级低于 history buffer”三点。总体可用，属于方向正确的概览文档。
