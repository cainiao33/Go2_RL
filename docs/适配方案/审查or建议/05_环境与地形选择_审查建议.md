# 05_环境与地形选择 审查/建议

## 总体评价

这份文档对 Standard 和 Track 的地形结构理解基本正确，推荐先 Standard 后 Track 的路线也合理。但文档里有几个需要修正的点：Track 子地形顺序不应假定固定；Track 长度、每段长度和 `episode_length_s=40` 需要结合平台实际评估配置；导航观测代码示例中的机器人位置/yaw 获取方式需要验证；评估配置建议中 `level` 和 `num_envs` 的组合需要注意覆盖不全的问题。

## 判断正确的部分

### 1. 先 Standard 再 Track 是正确策略

Standard 训练能获得稳定基础运动能力，包括：

- 上坡
- 下坡
- 上楼梯
- 下楼梯
- 简单避障/迷宫适应

Track 是这些能力的串联，并额外要求导航。因此先训 Standard 再训 Track 是低风险路线。

### 2. maze 比例不宜过高

maze 需要导航/避障，reward 信号更稀疏。如果前期 maze 比例太高，容易拖慢基础步态学习。因此文档建议 10~15% 是合理的。

### 3. 楼梯类地形应重点关注

冠军方案中楼梯比例较高，文档据此建议提高 `pyramid_stairs` 和 `pyramid_stairs_inv` 比例是合理的。上下楼梯确实更容易触发摔倒、关节极限和姿态异常。

## 需要修正的重点问题

### 1. Track 子地形顺序不应写死为一种

文档写：

```text
start → slope → slope_inv → stairs → stairs_inv → open_entry_maze → end
```

赛题文档说明：Track 前段是 `pyramid_slope`、`pyramid_slope_inv`、`pyramid_stairs`、`pyramid_stairs_inv` 的任意排列，末段必须是 `open_entry_maze`。

因此应改成：

```text
start → 任意排列的 4 种子地形 → open_entry_maze → end
```

训练时不能只训练一种固定顺序，否则可能对评估顺序过拟合。

### 2. Track 配置中的 `sub_terrains` 不能重复，但顺序要多样化

赛题文档评估示例中提到：

```text
sub_terrains 长度必须等于 track_length，不可重复
```

因此训练可以做多阶段任务，每次换一种排列，例如：

```toml
# Track-A
sub_terrains = ["pyramid_slope", "pyramid_slope_inv", "pyramid_stairs", "pyramid_stairs_inv", "open_entry_maze"]

# Track-B
sub_terrains = ["pyramid_stairs", "pyramid_slope", "pyramid_stairs_inv", "pyramid_slope_inv", "open_entry_maze"]

# Track-C
sub_terrains = ["pyramid_slope_inv", "pyramid_stairs_inv", "pyramid_slope", "pyramid_stairs", "open_entry_maze"]
```

建议文档补充“多排列轮训”。

### 3. “每段 8m，Track 40s”需要谨慎

Standard 走穿判定中默认地形长度 `Lterrain=8m`，但 Track 模式中每段子地形是否都按 8m 计算，需要平台环境确认。`episode_length_s=40` 是合理假设，但不是硬结论。

建议写成：

> Track episode_length_s 初始建议 35~40s，若 timeout_count 高但 abnormal_count 低，应优先增大 episode_length_s 或降低速度命令难度。

### 4. Track 导航观测代码中 yaw 获取方式不成立

文档中写：

```python
yaw = ...  # 从 projected_gravity 或其他方式获取
```

`projected_gravity` 可以反映 roll/pitch，但无法唯一恢复 yaw。更好的方式：

- 优先使用环境提供的 yaw/heading 信息（如果有）
- 如果只有 `goal_yaw` 而没有 robot yaw，需要从 root quaternion 获取
- 如果无法获取 robot yaw，则使用世界坐标下 goal delta 的归一化方向，先不转本体系

第一版建议用更稳的特征：

```text
[dx_world, dy_world, distance, dx_norm, dy_norm]
```

这样不依赖 robot yaw。

### 5. `env.scene["robot"].data.root_pos_w` 需要验证

平台文档只明确说 Track 地形额外提供：

- `env.goal_positions`
- `env.goal_yaw`
- `env.scene.sensors["nav_scanner"]`

没有明确机器人 root position 的访问路径。当前示例中使用 `env.scene["robot"].data.root_pos_w` 可能是 Isaac Lab 常见写法，但不保证在平台封装中可用。

建议：

- 在代码中先打印 `dir(env)`、`dir(env.scene)` 验证
- 或从现有 obs/infos 中找 robot position
- 不确认前不要把该代码作为正式方案

### 6. 评估 `num_envs=16` 只适合看视频，不适合统计

文档建议评估用 `num_envs=16` 产出 mp4，这适合可视化，但统计不稳定。建议分两类评估：

| 类型 | num_envs | 目的 |
|------|----------|------|
| 可视化评估 | 16 | 看视频、定位失败原因 |
| 统计评估 | 256~1024 | 看平均分、完成率 |

平台文档说 `num_envs <= 16` 才产出 mp4，不代表正式评估必须 16。

### 7. Standard 初期完全去掉 maze 有利有弊

初期 `maze=0.0` 有助于基础 locomotion，但如果后续 Track 终点固定是 maze，完全不训练 maze 会让 Track 初期困难。建议：

- 前 20~30% 训练：maze = 0.0~0.05
- 中期：maze = 0.10
- Track 前预热：maze = 0.15~0.20

这样不会突然引入迷宫难度。

## 建议版地形训练计划

### Stage A：基础地形

```toml
pyramid_slope = 0.25
pyramid_slope_inv = 0.25
pyramid_stairs = 0.25
pyramid_stairs_inv = 0.25
maze = 0.0
max_init_terrain_level = 3
```

### Stage B：加入迷宫

```toml
pyramid_slope = 0.20
pyramid_slope_inv = 0.20
pyramid_stairs = 0.25
pyramid_stairs_inv = 0.25
maze = 0.10
max_init_terrain_level = 5
```

### Stage C：Track 预热

```toml
pyramid_slope = 0.15
pyramid_slope_inv = 0.15
pyramid_stairs = 0.25
pyramid_stairs_inv = 0.25
maze = 0.20
max_init_terrain_level = 7
```

### Stage D：Track 多排列轮训

每次训练任务使用不同 `sub_terrains` 排列，均以 `open_entry_maze` 结尾。

## 建议补充的决策规则

```text
如果 abnormal_count 高：降低难度/降低 push/加强姿态稳定
如果 timeout_count 高：提高速度奖励/延长 episode/降低地形难度
如果 energy_score 低：后期增强 torque/action_smoothness 惩罚
如果 pose_score 低：增强 flat_orientation，但不要影响坡面通过
如果 Track 完成率低：优先检查 goal feature 和 navigation_progress 是否有效
```

## 结论

这份文档整体可用，但需要修正三点：**Track 顺序不能写死、导航观测中的 yaw/root position 访问需验证、评估要区分视频评估和统计评估**。建议新增“Track 多排列轮训”和“可视化/统计双评估”两节，使环境策略更稳健。
