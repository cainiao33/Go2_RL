# 03_技术方向选型 审查/建议

## 总体评价

这份文档的主结论是正确的：本届赛题优先做纯 PPO，而不是直接复刻冠军的 CTS/MoE。原因在于当前平台 policy 观测已经包含 `height_scan(256)`，不再是冠军方案中典型的 blind student 场景。因此，CTS 的边际收益下降，实施成本却很高。

## 判断正确的部分

### 1. PPO 作为主线是正确选择

当前代码包已有完整 `agent_ppo`：

- Actor-Critic
- RolloutStorage
- PPO loss
- GAE
- adaptive KL
- 模型保存/加载
- 平台 workflow 接入

因此第一阶段不应从 `agent_diy` 重写算法，而应在 `agent_ppo` 上做：

- reward 调优
- observation 扩展
- config 分阶段
- monitor 补充

### 2. 对 CTS 动机的重新审视是合理的

文档指出：冠军方案 CTS 的 student 主要依靠 45 维 proprio + history，而本届 policy 已经有 301 维 obs，其中包含地形扫描。这是关键差异。

因此：

```text
冠军 CTS：Teacher 帮 blind Student 学地形
本届平台：Policy 已经直接看 height_scan
```

这个判断成立。

### 3. MoE 不建议早做是正确的

MoE 的复杂度主要在：

- expert 分工
- router 训练
- loss 设计
- 训练不稳定
- 平台调试困难

在比赛周期有限的情况下，MoE 的性价比低于 reward/curriculum。

## 需要补充或修正的点

### 1. “PPO 足够”不代表不需要时序信息

当前 baseline 是单帧 MLP。虽然 policy 有 height_scan，但仍缺少显式历史信息，例如：

- 真实运动速度趋势
- 关节加速度趋势
- 动作延迟下的动态响应
- 地形接触后的稳定恢复

因此文档最后提出 history buffer 是对的，但建议把它从“折中方案”提升为 **PPO 主线的 P2 优先级**。

推荐路线应改为：

```text
PPO-MLP baseline → PPO + goal features → PPO + history buffer/GRU → 再考虑 CTS
```

### 2. CTS 不一定必须放在 `agent_diy`

文档写“需要在 `agent_diy` 框架下实现双模型训练”。这不一定是唯一方案。

也可以：

- 在 `agent_ppo` 下扩展 `ActorCritic` 为 teacher/student 双网络
- 在 `AlgorithmPPO` 中增加 distillation loss
- 保持现有 workflow 不大改

但实际可维护性上，仍建议不要早做 CTS。

建议改成：

> CTS 可在 `agent_ppo` 中增量扩展，也可在 `agent_diy` 中重写；但两者都需要较多调试，优先级低于 PPO + history。

### 3. 对 CTS 收益预估要更保守

文档写“相比纯 PPO 提升 5~15%”。这个数值没有足够证据支撑，尤其本届 policy 已有 height_scan。建议改成：

> CTS 的收益不确定，可能提升复杂地形稳定性，也可能因为训练目标冲突导致收益很小。只有在 PPO + history 到达瓶颈后再尝试。

### 4. History Buffer 的实现方式需考虑维度爆炸

文档建议：

```text
301 × history_len
```

如果 `history_len=5`，输入维度为 1505；如果 Track 再加 goal feature，会更大。MLP 第一层参数量显著增加，训练更慢。

建议两种更稳实现：

#### 方案 A：只对 proprio 做 history

```text
history_proprio = 45 × 5 = 225
current_height_scan = 256
current_goal = 5
final_obs = 225 + 256 + 5 = 486
```

优点：保留时序动态，避免对 height_scan 做冗余历史堆叠。

#### 方案 B：GRU 只处理 proprio

```text
GRU(proprio_history) → latent(128)
concat(latent, current_height_scan, goal_features)
```

优点：更合理，但实现复杂一些。

**建议第一版用方案 A。**

### 5. Track 导航不应只归类为 PPO 问题

Track 模式更像“低层运动控制 + 高层导航目标”的混合任务。可以考虑两种路线：

| 路线 | 说明 | 推荐度 |
|------|------|--------|
| 端到端 PPO | obs + goal + height_scan → 12 维动作 | 高，先做 |
| 隐式高层命令 | 根据 goal 生成速度 command，低层继续跟踪 command | 中，需要能控制 command 来源 |
| 显式路径规划 | 利用 nav_scanner 做规则规划 | 低，平台接口不确定 |

当前文档默认端到端 PPO，这合理，但应补充：如果能改 command manager 或 policy obs，goal-relative command shaping 是重要方向。

## 建议后的技术路线

### 主线：低风险路线

```text
1. PPO-MLP baseline 跑通 Standard
2. Standard reward/curriculum 调优
3. Track 加 goal features + dense navigation reward
4. PPO + proprio history buffer
5. checkpoint 筛选 + 能耗/姿态精调
```

### 支线：中风险路线

```text
1. PPO + GRU / temporal encoder
2. teacher 用 critic_obs 辅助训练 student
3. distillation loss 只作为辅助项，不替代 PPO loss
```

### 暂缓路线

```text
MoE / 多专家 / router / MCP / 复杂 CTS
```

## 最终建议

- **保留文档主结论**：PPO 是第一优先级，CTS/MoE 不应早做。
- **提高 history buffer 优先级**：建议作为 PPO 主线增强，而不是 CTS 的替代说明。
- **修正 CTS 收益表述**：不要给明确 5~15% 收益承诺。
- **避免全量 301 维 history 堆叠**：优先做 proprio history + current height scan。

## 结论

这份技术选型文档方向正确。建议把技术路线明确收敛为：**PPO baseline → reward/curriculum → Track goal features → proprio history → 再考虑 CTS**。这样更符合当前平台约束和比赛时间成本。
