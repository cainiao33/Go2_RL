# 04_训练策略规划 审查/建议

## 总体评价

这份训练策略文档已经围绕评分公式展开，方向正确：Standard 先走远，Track 先完成。但 reward 权重、域随机化强度、训练阶段超参里有一些偏激进或需要接口验证的内容。建议把方案拆成“第一版可执行策略”和“第二版增强策略”，避免一开始引入太多不确定项。

## 主要正确点

### 1. 对评分公式的理解正确

Standard：

```text
0.4 距离 + 0.2 时间 + 0.2 能耗 + 0.2 姿态
```

文档把距离放第一优先级是正确的，因为没走穿时，时间分难以获得，整体分数上限低。

Track：

```text
完成率 × (0.4 时间 + 0.4 姿态 + 0.2 能耗)
```

文档把完成率放第一优先级是正确的，因为完成率是乘数。

### 2. 分阶段训练合理

- Standard locomotion
- Track navigation
- 后期精调能耗/姿态

这个顺序符合任务结构。

### 3. 训练后期降低随机化强度的思路正确

评估时随机化和噪声默认关闭，因此训练后期从强随机化切到弱随机化，有助于模型在评估环境中表现更流畅。

## 需要修正的高风险点

### 1. Reward 权重建议不要一次性大改

文档建议：

```text
track_lin_vel_xy = 1.5
flat_orientation = -2.0
dof_pos_limits = -5.0
joint_torques = -2e-4
reach_goal = 30.0
```

这些方向合理，但直接大幅修改可能导致：

- 模型过度保守，不敢走快
- 大力矩惩罚过强导致爬楼梯能力下降
- 姿态惩罚过强导致在坡面/楼梯上无法接受必要姿态变化
- reach_goal 稀疏奖励过强但前期到不了，几乎无训练信号

建议采用渐进策略：

| 阶段 | 调参策略 |
|------|----------|
| 第一次跑通 | 使用默认 reward，仅小幅改动 |
| 稳定行走后 | 增加姿态/能耗权重 |
| 能走穿后 | 增加时间/速度驱动 |
| Track 有完成样本后 | 提高 reach_goal 权重 |

### 2. `flat_orientation` 权重不宜过大

赛题姿态分基于 roll/pitch 偏移，确实重要。但复杂坡面和楼梯上，机器人需要一定姿态调整。过大的 `flat_orientation` 可能导致：

- 上坡时身体姿态调整不足
- 下楼梯时不敢俯仰缓冲
- 跨障碍时动作僵硬

建议初始：

```text
flat_orientation = -1.5  # 默认
```

后期再试：

```text
flat_orientation = -2.0 ~ -3.0
```

### 3. 能耗惩罚应后期再增强

`joint_torques` 和 `joint_acc` 对能耗分有帮助，但在前期过强会阻碍模型学会爬坡/爬楼梯。建议：

```text
前期：joint_torques = -1e-4
中期：joint_torques = -1.5e-4
后期：joint_torques = -2e-4
```

### 4. `feet_air_time` 需要确认可实现

文档把 `feet_air_time` 作为新增 reward，但它通常需要：

- feet contact forces
- feet indices
- last_contacts
- feet_air_time buffer

当前平台文档没有明确说明 `RewardProcessBase` 中是否暴露这些。建议将其列为 P2/P3，而不是第一版必做。

### 5. Soft Curriculum 示例中可能访问不到 `base_lin_vel`

示例中写：

```python
self.base_lin_vel
self.commands
```

但 policy obs 中没有 base_lin_vel，critic_obs 才有 base_lin_vel。RewardProcess 是否可直接访问这些环境状态需要确认。建议第一版不要覆盖内置 `track_lin_vel_xy`，而是在已有 reward 基础上新增辅助项。

### 6. friction_range 和 push 强度偏激进

文档建议：

```text
friction_range = [0.2, 2.5]
push_interval_s = 8
max_push_vel_xy = 1.5
```

这对初期训练过强，可能导致大量摔倒，训练不稳定。建议分阶段：

| 阶段 | friction_range | push_interval_s | max_push_vel_xy |
|------|----------------|-----------------|-----------------|
| 初期 | [0.5, 1.2] | 20 | 0.3 |
| 中期 | [0.3, 1.5] | 15 | 0.5 |
| 强化 | [0.2, 2.0] | 10 | 1.0 |
| 后期评估适配 | [0.5, 1.2] | 关闭或 0.3 | 0.0~0.3 |

### 7. `reward_mean` 不一定等价于得分

文档把 `reward_mean` 持续上升作为训练健康度，这有参考价值，但赛题文档明确：reward 不等于平台 score。应优先看：

Standard：

- `total_score`
- `distance_score`
- `time_score`
- `energy_score`
- `pose_score`
- 各地形分项 score

Track：

- `completed_count`
- `abnormal_count`
- `timeout_count`
- `total_score_track_l{0~9}`
- `time_score_track_l{0~9}`
- `pose_score_track_l{0~9}`

## 建议版训练策略

### Phase 0：baseline 校准

```text
目标：确认默认 PPO 能正常学习
配置：默认 reward + 默认 domain_rand
关注：是否出现 NaN、abnormal_count 是否过高
```

### Phase 1：Standard 稳定通过

```text
目标：distance_score 提升，completed_count 增加
动作：略提高 track_lin_vel_xy，保持能耗惩罚默认
地形：先不加或少加 maze
```

### Phase 2：Standard 多地形强化

```text
目标：各地形 distance_score 接近
动作：加入 maze 10%，增强楼梯比例
随机化：恢复默认或略增强 friction/push
```

### Phase 3：Track 导航

```text
目标：形成完成样本
动作：加入 goal feature + navigation_progress + reach_goal
注意：reach_goal 不要单独依赖，必须配 dense progress reward
```

### Phase 4：评分精调

```text
目标：提高 energy_score 与 pose_score
动作：提高 joint_torques/action_smoothness/flat_orientation
随机化：降低，贴近评估环境
```

## 推荐的第一版 reward 改动

不要一次性加太多。建议第一版只改：

```text
Standard:
- track_lin_vel_xy: 1.0 → 1.2
- flat_orientation: -1.5 保持或 -1.8
- joint_torques: -1e-4 保持
- action_rate: -0.01 保持

Track:
- reach_goal: 10.0 → 15.0
- 新增 navigation_progress: 2.0~3.0
- 新增 goal_direction: 1.0~2.0
```

## 结论

该文档的训练方向正确，但第一版执行时应更保守。建议把强随机化、强能耗惩罚、feet_air_time、动态 sigma 放到后续迭代；先用默认 PPO + 小幅 reward 调整跑出稳定基线，再逐步引入冠军方案中的复杂技巧。
