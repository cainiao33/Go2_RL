# 技术方向选型：PPO vs CTS vs MoE

---

## 1. 冠军方案支持的三条技术路线

| 路线 | 核心思想 | 复杂度 | 冠军使用情况 |
|------|----------|--------|-------------|
| **PPO** | 标准 Actor-Critic，端到端训练 | ★☆☆ | 基础版本 |
| **CTS** | Teacher（特权 obs）→ Student（受限 obs）蒸馏 | ★★☆ | 主力方案 |
| **MoE-CTS** | 多专家路由 + Teacher-Student | ★★★ | 进阶变体 |

---

## 2. 本届赛题下各路线分析

### 2.1 纯 PPO（推荐首选）

**适用性：★★★★★**

理由：
- 本届平台的 policy obs **已经包含 256 维 height_scan**
- 冠军 CTS 的核心动机是：student 是 blind（仅 45 维本体感知），需要 teacher 给的地形知识
- 但现在 student 已经**不是 blind** 了，它直接看到了地形！
- 因此 CTS 的"蒸馏地形知识"动机大幅减弱

**优势**：
- 实现简单，直接在 baseline 上改
- 调试方便，不需要管理 teacher/student 两套模型
- 平台约束少（只需一套 workflow）

**劣势**：
- 可能缺少冠军方案中 teacher 的额外特权信息（foot contact forces、motor torques、motor acc）
- 泛化能力可能稍弱（但平台域随机化本身就有限）

**结论：本届赛题下，纯 PPO + 精心设计的 reward/curriculum 是性价比最高的方案。**

---

### 2.2 CTS 蒸馏（可选第二阶段）

**适用性：★★★☆☆**

**动机重新审视**：

冠军方案 CTS 的逻辑：
```
Teacher: 看 privileged obs (263维: base_lin_vel + contact + torques + heights)
Student: 只看 proprio (45维) + history buffer
→ Student 通过蒸馏学会 "从历史推断地形"
```

本届平台的情况：
```
Policy: 已经看 obs (301维: proprio + height_scan)
Critic: 看 privileged obs (316维: + base_lin_vel + joint_effort)
→ Policy 已经有地形信息了！
```

**如果仍要用 CTS，需要重新定义"特权"**：
- Teacher 额外看：`base_lin_vel`、`joint_effort`、`foot_contact_forces`、`goal_positions`（track 模式）
- Student：看标准 301 维 obs
- 蒸馏目标：让 student 学会从 height_scan 推断更多信息

**实现难度**：
- 需要在 `agent_diy` 框架下实现双模型训练
- workflow 需要自己管理 teacher/student 的分离与蒸馏 loss
- 模型保存时只保存 student

**收益预估**：相比纯 PPO 提升 5~15%，主要体现在复杂地形稳定性。

---

### 2.3 MoE（不推荐本届使用）

**适用性：★★☆☆☆**

理由：
- MoE 的核心是针对**不同地形/速度区间**训练专门的 expert
- 本届地形种类有限（5 种），标准 MLP 足以覆盖
- MoE 实现复杂度极高，需要自定义 router + 多 expert head
- 平台调试困难，无法本地跑 Isaac Gym 验证
- 性价比不如把时间花在 reward 工程和课程调参上

**仅当 PPO + curriculum 方案遇到明显瓶颈时，再考虑 MoE。**

---

## 3. 推荐决策树

```
开始
 │
 ├─ Phase 1: 纯 PPO + reward 工程 + 课程学习
 │   └─ 目标: Standard 全地形通过率 > 80%
 │
 ├─ Phase 2: 加入 Track 导航（goal obs + nav reward）
 │   └─ 目标: Track 完成率 > 60%
 │
 ├─ [可选] Phase 3: 如果 Standard 遇到瓶颈 → 尝试 CTS
 │   └─ Teacher 用 critic_obs + env 特权信息
 │   └─ Student 用标准 policy obs
 │   └─ 蒸馏 loss = KL(teacher_action_dist || student_action_dist)
 │
 └─ [不推荐] MoE → 除非有大量时间且前面方案已饱和
```

---

## 4. 各路线时间投入估算

| 路线 | 实现时间 | 调试时间 | 预期收益 | 风险 |
|------|----------|----------|----------|------|
| PPO + reward/curriculum | 3~5 天 | 5~7 天 | 高 | 低 |
| + Track 导航 | 2~3 天 | 3~5 天 | 中高 | 中 |
| + CTS 蒸馏 | 5~7 天 | 7~10 天 | 中 | 中高 |
| + MoE | 10+ 天 | 10+ 天 | 低~中 | 高 |

---

## 5. 最终推荐

> **以纯 PPO 为主干，重点投入 reward 工程和课程学习。**
> 冠军方案的核心竞争力不是 CTS/MoE 架构本身，而是他们丰富的 reward 设计（dynamic sigma、action smoothness、feet air time、hip regularization）和精细的课程策略（command range curriculum、zero command curriculum）。
> 这些经验可以在纯 PPO 框架下完全复现。

## 6. History Buffer 折中方案（推荐）

冠军 CTS 的 student 使用了 history buffer（多帧历史观测拼接）。即使不做 CTS，也可以借鉴：

```python
# 在 observation_process 中维护历史
self.history = torch.zeros(num_envs, history_len, obs_dim)

def process(self):
    obs = self.default_observation()
    self.history = torch.cat([self.history[:, 1:], obs.unsqueeze(1)], dim=1)
    return self.history.flatten(1)  # history_len × obs_dim
```

- 增加模型输入维度：`301 × history_len`
- 或使用 1D CNN / GRU 处理历史序列
- 好处：隐式估计速度/加速度趋势，提升地形适应性
- 代价：模型变大，训练变慢

**建议 history_len = 3~5，配合 1 层 GRU 或直接 flatten + MLP。**
